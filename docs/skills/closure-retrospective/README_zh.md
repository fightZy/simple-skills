# Closure Retrospective

本文档面向这个 skill 的维护者，不用于运行时路由。运行时入口仍然是 `SKILL.md`。

## 这个 Skill 是什么

`closure-retrospective` 是一个轻量的收尾复盘 skill，用来判断一个任务结束前是否出现了值得固化的可复用经验。

它的目标是让 agent：

- 在任务收尾阶段识别反复出现的纠正模式和流程摩擦
- 把可复用经验和一次性插曲区分开
- 为建议选择正确的落点
- 在修改任何 skill 或指令文件前先征求用户确认

## 用途

当一个非平凡任务已经接近完成，并且可能出现了值得保留的经验时，使用这个 skill。

这个 skill 不负责：

- 项目记忆沉淀
- 普通收尾总结
- 未经确认直接改规则

## 设计原则

- 先闭环再复盘：核心工作没收住前，不运行它。
- 默认保守：弱信号不应该生成新规则。
- 先建议后修改：第一轮只提出建议，不直接落文件。
- 选最窄的正确目标：能补已有 artifact 就不要新增。
- 净收益过滤：只有长期收益大于维护负担时才沉淀。

## 触发模型

运行时只有同时满足下面两点才应触发：

- 任务已经进入最终交付阶段
- 当前任务里确实出现了可复用经验，而不只是一个有意思的观察

常见信号：

- 用户多次纠正同类问题
- 工具或命令反复产生摩擦
- 本可避免的流程回环
- 某条判断规则明确改善了结果

## 落点选择

应当选择最窄、最合适的目标：

- workflow 局部规则进已有 skill
- 独立可复用流程才考虑新 skill
- 仓库级默认行为或约束才进 `AGENTS.md` 或 `CLAUDE.md`

如果这些目标都不明显成立，就应建议不做修改。

## 验证

好的验证场景应包含几类容易诱导过度沉淀的收尾任务：

- 只有一个有趣但很窄的问题的任务
- 有重复纠正，确实值得建议加规则的任务
- 根本不该输出复盘建议的任务

好的输出应体现：

- 明确引用当前任务中的证据
- 能说清为什么这条经验可复用
- 目标选择正确
- 行为是先请求确认，而不是静默改文件
